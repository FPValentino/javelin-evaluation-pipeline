# Javelin Evaluation Pipeline

An automated statistical evaluation suite for comparing the standard **Ochiai** fault localization algorithm against the enhanced **Ochiai-MS** (Mutation Score) algorithm. 

This pipeline is designed to process output CSVs generated by the Javelin IntelliJ IDE extension running against the GitBug-Java benchmark. It calculates EXAM scores, Top-N accuracy, and performs a Wilcoxon Signed-Rank Test to determine statistical significance, outputting a formatted multi-sheet Excel report.

## 📂 Repository Structure

* `generate_patches.py`: Compares the buggy and fixed source code folders to generate `.patch` files.
* `build_ground_truth.py`: Parses the `.patch` files to extract the exact modified line numbers and dynamically builds the Answer Key.
* `Evaluation Script.py`: The core interactive evaluation engine. It reads the ground truth JSON, batch-processes the CSVs, calculates metrics, and exports statistical comparisons.
* `ground_truth.json`: The mapping file defining the actual location of the bug (`FullyQualifiedClass` and `LineNumber`) for each evaluated program.
* `ochiai_results/`: Directory containing the raw statement-level CSV outputs from the standard Ochiai algorithm.
* `ochiai_ms_results/`: Directory containing the raw statement-level CSV outputs from the Ochiai-MS algorithm.

## ⚙️ Prerequisites

This pipeline is optimized for **Python 3.11**. Ensure you have the following data science and utility libraries installed:
```bash
pip install pandas scipy tqdm unidiff openpyxl
```
*(Note: `openpyxl` is strictly required to generate the final formatted Excel data reports).*

## 🚀 How to Use

### 1. Generate Patches (The Diff)
Ensure you have extracted the `-buggy` and `-fixed` workspaces using the tools from the `javelin-env-setup` repository. Run the patch generator to compare them and create the diffs. The script will ask for your WSL username to automatically locate your directories:

```bash
python generate_patches.py
```

### 2. Build the Ground Truth (The Answer Key)
Run the extractor to find the exact faulty line numbers from the patches. This will automatically generate your `ground_truth.json` file:
```bash
python build_ground_truth.py
```

### 3. Generate Javelin CSVs
Open the buggy workspaces in IntelliJ and run the Javelin extension. Save the output CSVs into the `ochiai_statement_results` and `ochiai_ms_statement_results` folders inside this repository. Make sure the CSV files are named using the exact Bug ID (e.g., `jsoup-1.csv`).

### 4. Run the Evaluation Suite
Execute the main interactive grading script to calculate the statistical significance:
```bash
python "Evaluation Script.py"
```
The script will display an interactive terminal menu. Select Option 1 to batch-process the CSVs. The system will display a live progress bar, print the project report cards to the terminal, and automatically generate a `Javelin_Evaluation_Report.xlsx` file containing all Bug-Level, Project-Level, and Global Wilcoxon statistics.

## 📊 Evaluation Metrics

This study utilizes three primary metrics to objectively measure the effectiveness of the fault localization algorithms:

### 1. Top-N Score
Measures the number of bugs where the actual faulty line of code appeared within the top 1, 3, 5, or 10 ranked suggestions. 
* **Purpose:** This is a direct measure of developer usability. If a bug is consistently ranked in the Top-5, a developer using the Javelin extension will find the fault almost immediately without scrolling through irrelevant code. **(Higher is better)**.

### 2. EXAM Score (Bug Score)
The EXAM score calculates the percentage of executable code a developer must inspect before finding the actual bug, assuming they read the code line-by-line starting from the most suspicious statement.

`EXAM Score = (Rank of the Faulty Statement) / (Total Number of Executable Statements)`

* **Purpose:** It standardizes the algorithm's performance across programs of vastly different sizes. **(Lower is better)**. For example, an EXAM score of 0.05 means the developer only needed to read 5% of the codebase.

### 3. Wilcoxon Signed-Rank Test
A non-parametric statistical hypothesis test used to compare two related samples (the EXAM scores of Standard Ochiai vs. Ochiai-MS for the exact same bugs).
* **Purpose:** It proves mathematical significance. If the resulting **p-value is < 0.05**, it definitively proves that the improvements offered by Ochiai-MS are statistically significant and not due to random variation or algorithmic noise.
