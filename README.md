# Javelin Evaluation Pipeline

An automated statistical evaluation suite for comparing the standard **Ochiai** fault localization algorithm against the enhanced **Ochiai-MS** (Mutation Score) algorithm. 

This pipeline is designed to process output CSVs generated by the Javelin IntelliJ IDE extension running against the GitBug-Java benchmark. It calculates EXAM scores, Top-N accuracy, and performs a Wilcoxon Signed-Rank Test to determine statistical significance.

## 📂 Repository Structure

* `generate_patches.py`: Compares the buggy and fixed source code folders to generate `.patch` files.
* `build_ground_truth.py`: Parses the `.patch` files to extract the exact modified line numbers and dynamically builds the Answer Key.
* `Evaluation Script.py`: The core evaluation engine. It reads the ground truth JSON, batch-processes the CSVs, calculates metrics, and outputs statistical comparisons.
* `ground_truth.json`: The mapping file defining the actual location of the bug (`FullyQualifiedClass` and `LineNumber`) for each evaluated program.
* `ochiai_results/`: Directory containing the raw CSV outputs from the standard Ochiai algorithm.
* `ochiai_ms_results/`: Directory containing the raw CSV outputs from the Ochiai-MS algorithm.

## ⚙️ Prerequisites

This pipeline is optimized for **Python 3.11**. Ensure you have the following data science and utility libraries installed:

```bash
pip install pandas scipy tqdm unidiff
```

## 🚀 How to Use

### 1. Generate Patches (The Diff)
Ensure you have extracted the `-buggy` and `-fixed` workspaces using the tools from the `javelin-env-setup` repository. Run the patch generator to compare them and create the diffs:
* **LOOK INSIDE THE CODE! Change [your-profile] to your javelin-workspace directory in Linux IN: workspace_dir = r"\\wsl.localhost\Ubuntu\home\[your-profile]\javelin-workspaces"**

```bash
python generate_patches.py
```

### 2. Build the Ground Truth (The Answer Key)
Run the extractor to find the exact faulty line numbers from the patches. This will automatically generate your `ground_truth.json` file:

**Install unidiff first**
```bash
pip install unidiff
```
***Replace [your-profile] with your directory FIRST in this part of the code: patch_dir = r"\\wsl.localhost\Ubuntu\home\paul\javelin-workspaces\gitbug_patches"!***
```bash
python build_ground_truth.py
```

### 3. Generate Javelin CSVs
Open the buggy workspaces in IntelliJ and run the Javelin extension. Save the output CSVs into the `ochiai_results` and `ochiai_ms_results` folders inside this repository. Make sure the CSV files are named using the exact Bug ID (e.g., `jsoup-1.csv`).

### 4. Run the Evaluation
Execute the main grading script to calculate the statistical significance:
```bash
python "Evaluation Script.py"
```
The script will display a live progress bar and instantly output the final averages, Top-N counts, and the Wilcoxon p-value.

## 📊 Evaluation Metrics

This study utilizes three primary metrics to objectively measure the effectiveness of the fault localization algorithms:

### 1. Top-N Accuracy
Measures how frequently the actual faulty line of code appears within the top 1, 3, 5, or 10 ranked results. 
* **Purpose:** This is a direct measure of developer usability. If a bug is consistently ranked in the Top-5, a developer using the Javelin extension will find the fault almost immediately without scrolling through irrelevant code.

### 2. EXAM Score
The EXAM score calculates the percentage of executable code a developer must inspect before finding the actual bug, assuming they read the code line-by-line starting from the most suspicious statement.

`EXAM Score = (Rank of the Faulty Statement) / (Total Number of Executable Statements)`

* **Purpose:** It standardizes the algorithm's performance across programs of vastly different sizes. A lower EXAM score is always better (e.g., an EXAM score of 0.05 means the developer only needs to read 5% of the codebase).

### 3. Wilcoxon Signed-Rank Test
A non-parametric statistical hypothesis test used to compare two related samples (the EXAM scores of Standard Ochiai vs. Ochiai-MS for the exact same bugs).
* **Purpose:** It proves mathematical significance. If the resulting **p-value is < 0.05**, it definitively proves that the improvements offered by Ochiai-MS are statistically significant and not due to random variation or algorithmic noise.
